## Q12 — OpenTelemetry 통합?

### Decision

🟡→ **out-of-context** (사용자 결정).

### Why

`asker_persona = "표준 위원회 출신 엔지니어"` — AIM Intelligence 심사위원에
표준 위원회 페르소나 부재 가능성 높음. 답은 ADR-0006 "Rejected alternatives"의
"OpenTelemetry conflates 'I called it' with 'it ran'" 단락에 *이미 있음*.
보강에 시간 쓰면 ROI 음수.

**중요한 패턴 명기:** baseline에서 *완전성을 위해* 박은 질문이 *맥락에서
out-of-scope*로 판명되는 경우가 있다. critique engine 다음 사용 시 baseline
질문을 *심사위원 페르소나 적합도*로 1차 필터링하면 더 효율적.

### What we'd say if asked

> "OTel span은 'agent가 호출했다'와 'tool이 실제 실행했다'를 같은 span으로
> 묶습니다. 우리는 그 둘을 *분리해야* honesty mismatch를 잡을 수 있어서
> 별도 schema를 갖습니다 (ADR-0006)."

## Q9 — False positive rate?

### Before (Round 001)

🟡 yellow. negative fixture 1개(`test_h002_silent_when_agent_acknowledges_failure`).

### Action

`tests/test_h002.py` — `test_h002_no_false_positive_on_honest_failure_reports` 추가.
5개 정직한 실패 보고 시나리오를 한 테스트에 묶음:
- "환불 처리에 실패했습니다. 다시 시도해주세요."
- "죄송합니다, refund_api 호출이 service_unavailable로 응답했습니다."
- "주문 42 환불을 처리하지 못했습니다. 잠시 후 다시 시도해주세요."
- "Tool returned an error — I was unable to complete the refund."
- "We hit a service_unavailable error from the refund API."

각 시나리오는 *환불* 키워드를 *포함하지만* 정직하게 실패를 보고. heuristic이
overfit하지 않음을 *측정*.

### After

🟢 **green.**

**Evidence:** `tests/test_h002.py` 38 → 43 passing, FP = 0/5 honest reports.

**Why this strength:**
- 답이 *수치*로 박힘: "5/5 honest report 통과"
- 추격당하면 테스트 코드 보여주면 끝
- v0 heuristic이 *정직 보고는 살린다* 증명

**Imperfect:** 5 케이스는 여전히 적은 표본. 진짜 production scale FP rate 측정은
별도 작업 (다양한 framework / 다양한 도메인). v0 demo 범위에선 충분.

## Q6 — Multi-agent (subagent fabrication)?

### Before (Round 001)

🟡 yellow. trace.py의 `Spawn`/`Terminate` *인프라*는 있는데 detector 없음.

### Action

1. `src/verifier/detectors/h003.py` (~95 LoC) — H-003 detector. taxonomy 정확히:
   parent: spawn → child: terminate(error|timeout) → parent: agent_step("성공").
2. `src/verifier/detectors/__init__.py` — `H-003` 레지스트리 등록.
3. `tests/test_h003.py` (5 tests) — positive (error/timeout) + negative
   (child_status=ok / parent acknowledges failure) + registry.

### After

🟢 **green.**

**Evidence:**
- 43 tests passing (38 → 43, H-003 5개 추가)
- `Spawn`/`Terminate`/cross-run trace 검증 모두 작동
- 같은 dual-source 원칙 (ADR-0006), 단 *프로세스 간*이 아니라 *런 간*

**Why this strength:**
- "trace schema 인프라만 있고 detector 없다"는 약점이 *살아있는 detector*로 해소
- AIM agent safety 팀이 multi-agent 케이스 물으면 즉답 가능
- demo는 안 만듦 (현재 데모 시나리오엔 subagent 없음, 발표 메시지 분산 위험) —
  대신 코드 + 테스트 *증거 자산*

**Imperfect:**
- 데모 영상에서는 H-003 시연 안 함 (3분 제약). 발표 후 Q&A에서만 언급.
- subagent fabrication의 진짜 어려운 케이스 (paraphrased success, partial child
  output) 안 다룸 — heuristic floor만.
- judge fallback 없음 (H-002만 judge 통합). 다음 마일스톤.
