## Q15 — 도메인별로 detector 다시 짜야 하나?

> "도메인별로 각 세팅이 되어있어야 제대로 로그가 나오는 거 아닌가? 그러면
> 도메인을 제한해야 하는가? 또는 보편적인 룰셋으로만 구성해야 하는가?"

### Answer

도메인 제한도, 보편 룰만도 아님. **구조는 generic, 어휘만 additive.**

**Layer별 도메인 의존성:**

| Layer | Generic? |
|---|---|
| Pattern (H-002 등 정의) | 🟢 yes |
| Comparison logic (detector body) | 🟢 yes |
| LLM judge (semantic) | 🟢 yes |
| Heuristic keyword set | 🟡 partial — language-aware |
| Tool log shape (`ToolLogEntry`) | 🟢 generic |

**Integration cost (additive, all-or-nothing 아님):**

| 무엇 | LoC |
|---|---|
| 기본 detector 그대로 사용 | 0 |
| 도메인 success keyword 추가 | ~10 |
| 도메인 custom detector | ~50 |
| 새 framework adapter | ~80–170 |

### Action

1. `docs/taxonomy.md` — "Domain scope — what is and isn't generic"
   섹션 신규 (Versioning 직전). 4-row layer table + cost table +
   slogan.
2. `README.md` — "Domain scope" 한 단락 (What we don't claim 직후).
   slogan 한 줄 + cost range + taxonomy 링크.

### After

🟡 **yellow.**

**Why yellow not green:**
- 답은 *논리적이고 문서화*됨
- 시연은 *없음* — refund 도메인만 보여줌. 의료/금융/법률 도메인 패턴 시연 없음
- 추격당하면 "그럼 의료 도메인 detector 보여주세요" → "코드는 generic, 같은
  H-002로 잡힙니다, 다만 의료 어휘 추가가 필요합니다" 답 가능 but 시연은 없음

**Strength rationale:**
- "도메인 제한해야 하나" 함정 즉답: *제한 안 함, 어휘만 확장*
- "all-or-nothing 통합" 함정 즉답: *0 → 170 LoC 점진적*
- Slogan ("teach us your domain" vs "point us at reality") 한 줄 강력

**Imperfect:**
- 시연 0개. 도메인 generality는 *주장*, 증거 없음
- 다른 도메인에서 *진짜* 작동하는지 검증 안 함 (가설)

### Remediation queue

- (선택) 다른 도메인 toy tool 1개 추가 (예: `db_query.py`) — generality *시연*
- 또는 README에 "tested on refund + payment lifecycle (proposed H-005)" 한 줄
