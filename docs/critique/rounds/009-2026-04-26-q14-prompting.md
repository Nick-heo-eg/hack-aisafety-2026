## Q14 — 이거 그냥 프롬프트 인젝션으로 Role 지정하면 되는 거 아닌가요?

> "거짓말하지 말고, 확인한 것만 말하라" 라고 시스템 프롬프트에 박으면 끝
> 아닌가요?

### Answer (3-layer)

**Layer 1 — 정보 문제, 의지 문제 아님.**
모델은 *무엇이 실제로 실행됐는지* 모른다. tool log / 외부 상태에 접근 권한
없음. "확인된 것만 말해라" 지시를 받아도 *모르는 상태에서 정직하게 틀린 말*을
한다.

**Layer 2 — 프롬프트는 이미 시도됐다.**
모든 production agent — Claude Code, Cursor, Replit Agent, GitHub Copilot —
시스템 프롬프트에 *이미* 비슷한 문구 박혀 있음:

> "Only report what you actually did. Do not fabricate tool results.
>  Acknowledge errors honestly."

그런데도 거짓 보고가 나옴. Air Canada 챗봇 사고, Replit DB 삭제 후 가짜 데이터
보고, Cursor 테스트 거짓 통과 보고 — **전부 시스템 프롬프트에 honesty 지시가
있는 상태에서 일어난 사고**.

**Layer 3 — AIM Stinger의 존재 자체가 증거.**
AIM이 prompt injection을 *공격*하는 회사 (Stinger). 그게 작동한다는 게
*프롬프트로 안 된다*는 증거. 우리는 *프롬프트가 깨진 후*를 본다.

### One-liner (발표용)

> **"If prompts worked, Stinger wouldn't exist."**

또는:

> **"Prompts change what the agent says. We verify what actually happened."**

### Evidence

- AIM Stinger product 페이지 (prompt injection 공격 자동화)
- 우리 기존 사고 사례 매핑 (`docs/devlog/0008-...md`) — 4 사례 모두 honesty
  지시 있는 상태에서 발생
- 우리 Q4 demo 결과 (`run_paraphrase_demo.py`) — 휴리스틱 0/4, 의미 검증 필요.
  *프롬프트는 기껏해야 휴리스틱 수준 정확도 보장* — 의미 보장 못 함.

### Strength rationale

🟢 **green.**

**Why:**
- 답이 *3-layer*로 견고. 각 layer가 *독립적으로* 추격 차단.
- AIM Stinger 인용이 *심사위원 자기 product*를 증거로 만듦 — 정면 답 가능.
- 사고 사례 매핑이 이미 devlog에 박힘 — 시연 자산.
- 발표 한 줄이 *기억에 남음* ("If prompts worked, Stinger wouldn't exist.").

**Why this is high blast:**
- AIM 심사위원 *거의 100% 묻는다*. 보안 회사 입장에서 "프롬프트 layer"는 본인
  business의 일부.
- 답 못 하면 우리 도구 *존재 이유*가 흔들림. 가장 본질적인 질문 중 하나.

**Imperfect:**
- "프롬프트로 *전혀* 안 되나?"는 양극단 답. 현실은 *프롬프트가 일부 줄여줌*.
  솔직한 답: "프롬프트는 *floor*를 올린다. 그래도 *floor 위에서* 거짓말이 일어
  난다. 우리는 그 위에서 잡는다." — 더 nuanced 답이 필요한 청중도 있을 것.
- 양극단 답으로 가도 발표 30초 이내에는 충분.
