# 0005 — Presentation structure (2026-04-26)

## What

- 발표 7장 구조 확정. 4(컨셉) + 1(포지션) + 1(데모) + 1(어태치먼트).
- 톤: 변주 C (도발/데모 톤). 한국어 메인 + 영어 부제 병기.
- 용어 사전 6개 박음 (한↔영). README/슬라이드/devlog 모두 동일 단어
  사용 강제.
- README 통째 재작성 — 컨셉/문제/포지션/방향/How/슬라이드 구조/용어
  순. 코드 짜기 전에 *나침반*이 박힘.

## Why

### 왜 5장이 아니라 7장인가

당신 원안은 *문제 → 왜 문제 → 방향 → How → 데모* 5장. 결정적 슬라이드
하나가 빠져있었다 — **포지션 ("이 자리는 비어있다").**

심사 = AIM Intelligence. Stinger/AIM Guard/AIM Red Team Suite 운영 중,
시리즈 A 100억, OpenAI/Anthropic/Meta 보안 파트너. 첫 질문이 *반드시*
"이거 우리(AIM)가 이미 하지 않나?" 가 됨. 슬라이드 3에서 *선제 차단*
안 하면 데모 5분 끝나고 무너짐.

```text
   슬라이드 3 없으면:
   1 hook → 2 문제 → 4 방향 → 5 how → 6 데모 → ❓ "근데 우리가 이미..."

   슬라이드 3 있으면:
   1 hook → 2 문제 → 3 "이 영역 비어있다" → 4 방향 → 5 how → 6 데모 → 신뢰
```

어태치먼트(7) 추가는 사용자 제안. 동의 — *과정의 정직성*을 보여주는
슬라이드는 honesty verifier 발표에 메타적으로 잘 어울림.

### 왜 피벗을 *드러내는* 어태치먼트인가

10:55 마크에 "⚠ 피벗" 을 *지우지 않고 강조*. 두 가지 이유:

1. **Git log는 어차피 보임.** 발표장에서 심사위원이 한 번 본다. 발표에
   "직선 타임라인" 보였다가 git에 ⚠ 있으면 "왜 숨겼나" 됨.
2. **메타 증명.** Honesty Verifier가 *honesty 원칙으로 만들어졌다.*
   해커톤에서 6커밋만에 피벗 인정한 건 — 정직한 작업 흐름의 증거.

### 왜 변주 C (도발 톤)인가

청중이 5분 안에 *기억해야 하는 한 문장*이 필요. 학술 톤(A)은 정확하지만
까먹기 쉽고, 엔지니어 톤(B)은 신뢰 있지만 임팩트 약함.

```
"AI는 모든 걸 다 했다고 말한다. 진짜로 했는지는 아무도 검증하지 않는다."
```

이 한 문장이 슬라이드 1의 *전부*. 청중이 발표 끝나고 옆 사람과 이야기할
때 *이 문장이 나오는 게* 발표 성공의 척도.

본문은 B(엔지니어) 톤으로 안정감, 마무리는 A(학술) 어휘 한 줄로 깊이 —
3-tone 구조. README에 그대로 박혀있음.

### 왜 한·영 병기인가

AIM은 한국 회사 (1차 청중 한국어), OpenAI/Anthropic 파트너 (2차 청중
영어 가능). 메인 폰트(한국어) + 약 80% 회색 영어 부제로 *시각 위계*를
잡음. 1:1 동등 병기는 둘 다 약해 보여서 피함.

용어 사전 6개를 미리 박은 이유: 슬라이드마다 다른 영어 단어 나오면
청중이 *"같은 걸 말하는 건가"* 의심. 한 번 박힌 매핑을 발표·코드·문서
모두에서 강제.

## Snags

- **타임라인을 솔직히 적으면 자랑 같음.** "⚠ 피벗" 마크 하나에 톤이
  결정됨. 큰 글씨 아니고, 빨간색 아니고 — 그냥 ⚠ 한 글자만. *드러내되
  강조하지 않음.*
- **README가 코드보다 *나침반*이 됨.** 보통 코드가 먼저고 README가
  사후. 이번엔 역순. 다음 detector 작성 시 "이게 슬라이드 N의 무엇을
  뒷받침하나?" 로 자기 검증 가능.
- **발표 직전 퍼블릭 전환을 *문서에 박아둠*.** 슬라이드 7에 GitHub URL
  쓰는 의미가 있으려면 prerequisite. 까먹지 않게 README Status 섹션에
  주의 박스로.
